# 新版会员管理系统产品文档

## 1. 项目背景

现有系统为 Windows XP 时代开发的本地会员管理软件，目录内包含 `永丰文体会员管理系统.exe`、`club.mdb`、`SysSet.xml` 等文件，具备典型的单机版、单目录运行特征。现计划开发新版系统，要求在 Windows 10、Windows 11 下稳定运行，延续本地化、简单可移植的使用方式，并支持将旧版 `.mdb` 数据迁移到新系统。

本次建设目标不是做复杂的联网 SaaS 系统，而是做一个适合门店现场使用、绿色免安装、复制目录即可运行的新版本会员管理程序。

## 2. 产品目标

### 2.1 核心目标

1. 在 Windows 10/11 上稳定运行，界面和操作明显优于旧系统。
2. 不依赖 `.NET` 运行环境，程序复制到新电脑后可直接打开使用。
3. 数据本地存储，部署简单，便于备份和迁移。
4. 支持从旧版 `club.mdb` 中迁移会员及相关业务数据。
5. 满足会员登记、消费累计积分、积分兑换赠礼、会员查询等核心业务。

### 2.2 成功标准

1. 新电脑无需额外安装运行时即可打开程序。
2. 程序目录可整体复制到 U 盘或其他电脑后直接运行。
3. 旧系统主要会员数据可一键导入或按步骤导入。
4. 店员可在 1 分钟内完成会员搜索、消费登记、积分兑换等日常操作。
5. 查询速度快，支持手机号、姓名、拼音首字母、模糊关键字搜索。

## 3. 建设范围

### 3.1 本期范围

1. 会员基础资料管理
2. 会员查询与检索
3. 消费登记与积分累计
4. 积分赠礼登记
5. 数据迁移导入
6. 本地数据备份与恢复
7. 积分规则和基础参数设置
8. 操作日志与基础报表

### 3.2 暂不纳入本期

1. 多门店联网
2. 微信/短信通知
3. 在线支付接口
4. 云端同步
5. 复杂营销活动引擎

## 4. 用户角色

### 4.1 店员

负责录入会员、查询会员、登记消费、登记积分兑换。

### 4.2 店长/管理员

负责系统参数维护、数据迁移、备份恢复、礼品维护、查看统计报表。

## 5. 核心业务需求

## 5.1 会员信息管理

系统需支持新增、编辑、查询、停用会员资料。

会员基础信息建议包含以下字段：

1. 会员编号
2. 姓名
3. 性别
4. 出生月日
5. 手机号
6. 备注
7. 创建时间
8. 状态

其中：

1. 手机号建议作为重要检索字段，但不强制唯一，以兼容历史脏数据。
2. 出生年月允许只录入“年-月”，不强制具体日期。
3. 系统应自动生成姓名拼音全拼和拼音首字母索引，用于快速搜索。

## 5.2 会员搜索

系统必须支持以下查询方式：

1. 按手机号精确查询
2. 按姓名模糊查询
3. 按拼音首字母查询
4. 按会员编号查询
5. 混合关键字模糊搜索

示例：

1. 输入 `138` 可搜索手机号包含 `138` 的会员
2. 输入 `张` 可搜索姓名中包含“张”的会员
3. 输入 `zs` 可搜索“张三”“赵四”等拼音首字母匹配会员

搜索结果应展示：会员编号、姓名、手机号、当前积分、累计消费额、最近消费时间。

## 5.3 消费登记与积分累计

系统需支持对会员进行消费录入，并按照规则自动增加积分。

消费登记字段建议包含：

1. 消费单号
2. 会员编号
3. 消费时间
4. 消费金额
5. 本次新增积分
6. 操作人
7. 备注

积分规则要求：

1. 支持按“每消费 X 元增加 1 积分”配置
2. 支持向下取整规则，例如消费 35 元，规则为 10 元积 1 分，则增加 3 分
3. 支持后续扩展为“固定倍数日”或“手工调整”

备注：

现有 `SysSet.xml` 中存在 `JPDJ=10` 参数，推测可能与旧系统积分或奖品规则有关。新版应预留兼容配置项，在详细设计和迁移联调阶段确认其实际业务含义。

## 5.4 积分赠礼

系统需支持使用会员积分兑换礼品，并保留完整记录。

赠礼登记字段建议包含：

1. 兑换单号
2. 会员编号
3. 兑换时间
4. 礼品名称
5. 礼品数量
6. 消耗积分
7. 操作人
8. 备注

业务规则：

1. 兑换时自动校验会员当前积分是否足够
2. 兑换成功后自动扣减积分
3. 所有积分增加和扣减都必须保留流水，不允许只改余额

## 5.5 数据迁移

已知具备 `.mdb` 数据迁移工具，因此新版系统需提供“导入入口”和“导入结果校验”能力，而不是要求人工重录。

迁移功能要求：

1. 支持导入旧版 `club.mdb` 数据
2. 支持查看导入前预检结果
3. 支持导入后查看成功条数、失败条数、失败原因
4. 支持重复导入保护，避免同一批数据重复写入
5. 支持保留旧系统会员编号或旧主键映射关系
6. 支持导出迁移日志，便于人工复核

建议迁移内容至少包括：

1. 会员基础信息
2. 积分余额或积分流水
3. 历史消费记录
4. 历史兑换记录
5. 基础系统参数

## 5.6 基础设置

管理员可维护以下参数：

1. 积分累计规则
2. 礼品目录
3. 操作员账户
4. 数据备份目录
5. 门店名称
6. 打印或导出基础模板

## 5.7 数据备份与恢复

为满足“简单可移植”要求，系统需内置本地备份与恢复功能。

要求如下：

1. 支持一键备份当前数据库到指定目录
2. 支持按日期生成备份文件
3. 支持从备份文件恢复
4. 恢复前需提示风险并自动生成恢复前备份

## 5.8 报表与统计

本期建议提供基础统计能力：

1. 会员总数
2. 本月新增会员数
3. 本月消费总额
4. 本月新增积分
5. 本月积分兑换次数
6. 会员消费排行

## 6. 非功能需求

## 6.1 操作系统兼容性

1. 必须支持 Windows 10、Windows 11
2. 建议支持 64 位系统为主，必要时兼容 32 位运行
3. 程序不依赖 `.NET Framework`、`.NET Runtime` 或 Java 运行环境

## 6.2 部署与可移植性

1. 支持绿色版部署
2. 程序目录复制到新电脑后即可运行
3. 所有核心文件保存在应用目录内或明确的数据子目录内
4. 不强依赖注册表

建议目录结构：

```text
MemberClub/
  MemberClub.exe
  data/
    club.db
    backup/
  config/
    app.ini
  logs/
```

## 6.3 性能要求

1. 会员数 1 万以内搜索结果应在 1 秒内返回
2. 常规消费登记、兑换登记应在 1 秒内完成
3. 启动时间建议小于 3 秒

## 6.4 数据安全

1. 核心业务采用事务写入，避免异常断电造成积分错误
2. 所有积分变动必须生成流水
3. 删除会员采用“停用”优先，避免误删历史数据

## 7. 推荐技术方案

## 7.1 总体建议

推荐采用：

1. **桌面程序框架：Tauri 2**
2. **本地数据库：SQLite**
3. **配置文件：INI 或 JSON**
4. **迁移工具对接：单独导入模块**

## 7.2 推荐原因

### Tauri 2

优点：

1. 编译后的 Windows 程序为原生可执行文件
2. 不依赖 `.NET`
3. 部署简单，适合绿色软件
4. 对老式 Windows 桌面业务软件风格适配较好
5. 文件体积和资源占用通常低于 Electron 类方案

### SQLite

优点：

1. 单文件数据库，便于备份和复制
2. 无需单独安装数据库服务
3. 稳定成熟，适合单机本地业务
4. 与绿色版程序天然匹配

## 7.3 不建议方案

1. 不建议使用依赖 `.NET` 的 WinForms/WPF/MAUI
2. 不建议使用必须安装数据库服务的 MySQL/PostgreSQL 单机方案
3. 不建议优先选择 Electron，除非后续有复杂界面需求，因为体积较大、分发较重

## 9. 数据迁移方案

## 9.1 迁移原则

1. 先迁移后启用，避免新旧系统同时改同一份数据
2. 保留旧系统主键映射，便于追溯
3. 优先保证会员主数据、积分余额、历史流水一致
4. 导入过程可重复执行，但同一批数据不能重复入库

## 9.2 迁移流程

1. 选择旧版 `club.mdb`
2. 系统执行预检
3. 生成字段映射结果和异常提示
4. 用户确认后执行正式导入
5. 导入完成后生成报告
6. 用户抽样核对会员、积分、消费、兑换数据

## 9.3 预检规则

1. 检查必填字段是否为空
2. 检查手机号格式是否合法
3. 检查姓名是否为空
4. 检查历史积分是否为负数
5. 检查重复会员编号
6. 检查消费金额和积分计算逻辑是否异常

## 9.4 迁移后的校验建议

1. 会员总数与旧系统对比
2. 会员积分余额抽样对比
3. 历史消费总笔数对比
4. 历史兑换总笔数对比
5. 随机抽查 20 个会员的完整历史

## 10. 界面建议

为了适应门店快速操作，界面应强调“少层级、快检索、少弹窗”。

建议首页功能：

1. 搜索会员
2. 新增会员
3. 登记消费
4. 积分赠礼
5. 数据迁移
6. 备份恢复
7. 统计报表

建议会员主界面布局：

1. 顶部放统一搜索框
2. 左侧显示会员列表
3. 右侧显示会员详情、积分、消费记录、兑换记录
4. 常用按钮固定显示，减少菜单层级

## 12. 验收标准

1. 在 Windows 10、Windows 11 实机上可直接运行
2. 在无 `.NET` 环境的电脑上可正常运行
3. 程序目录复制后可直接启动
4. 能成功导入旧版 `.mdb` 数据
5. 会员可按姓名、手机号、拼音首字母进行搜索
6. 消费后积分自动增加且流水正确
7. 积分兑换后余额正确扣减且流水正确
8. 可完成备份和恢复

## 13. 风险与注意事项

1. 当前尚未直接读取旧版 `club.mdb` 表结构，正式开发前需结合迁移工具确认旧表字段和数据质量。
2. 历史数据可能存在手机号缺失、重复会员、姓名不规范等问题，迁移方案必须支持脏数据处理。
3. 若旧系统积分规则不止一种，需先梳理历史规则，再确定导入后余额和流水生成策略。
4. 若后续需要联网、多门店或扫码设备对接，当前单机架构需提前留扩展接口。

## 14. 建议实施步骤

1. 先确认旧版数据库表结构和迁移工具输入输出格式
2. 确认积分规则、旧参数 `JPDJ` 的真实业务含义
3. 完成新版数据库设计和原型界面
4. 开发 P0 功能
5. 联调迁移工具并进行历史数据试迁移
6. 进行门店试运行
7. 根据试运行结果优化后正式替换旧系统
