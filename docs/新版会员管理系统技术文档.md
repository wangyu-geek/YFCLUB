# 新版会员管理系统技术文档

## 1. 文档说明

### 1.1 编写目的

本文档依据《新版会员管理系统产品文档》整理，目标是将产品需求转换为可实施的技术方案，供研发、测试、实施和后续维护使用。

### 1.2 文档范围

本文档覆盖以下内容：

1. 技术架构与部署方案
2. 功能模块拆分与职责边界
3. 数据库模型与关键索引设计
4. 核心业务流程与事务控制
5. 旧版 `club.mdb` 数据迁移与幂等设计
6. 备份恢复、日志审计、性能与安全设计
7. 测试策略与验收映射

### 1.3 设计依据

1. 产品文档中已明确的业务范围、角色与验收标准
2. 现有系统目录结构：`永丰文体会员管理系统.exe`、`club.mdb`、`SysSet.xml`
3. 已确认旧配置文件 `SysSet.xml` 中存在参数 `JPDJ=10`

### 1.4 当前约束与待确认项

1. 当前尚未直接确认旧版 `club.mdb` 的真实表结构与字段含义，迁移映射以“可扩展导入模块”方式设计。
2. 本机环境未发现可直接读取旧版 `.mdb` 的 Jet/ACE OLEDB 驱动，因此迁移设计默认通过独立导入模块或外部迁移工具完成。

## 2. 建设目标与技术原则

### 2.1 建设目标

1. 提供可在 Windows 10/11 上稳定运行的绿色版单机会员管理系统。
2. 保证核心业务闭环：会员管理、搜索、消费积分、积分兑换、数据迁移、备份恢复。
3. 保证核心数据可追溯、可恢复、可审计。
4. 支持旧系统历史数据平滑迁移，避免同批数据重复入库。

### 2.2 技术原则

1. 单机优先：不引入数据库服务、注册表依赖和复杂部署流程。
2. 事务优先：积分余额、积分流水、消费记录必须在同一事务内提交。
3. 幂等优先：迁移、导入、恢复等高风险操作必须具备重复执行保护。
4. 可追溯优先：所有关键业务动作保留业务流水和操作日志。
5. 易维护优先：目录清晰、配置简单、日志可导出、数据库可直接备份。

## 3. 总体技术架构

### 3.1 技术选型

推荐采用以下实现组合：

1. 桌面容器：Tauri 2
2. 后端语言：Rust
3. 前端：Vue 3 + TypeScript + Vite
4. 本地数据库：SQLite
5. 数据访问：`rusqlite` 或等价 SQLite Rust 封装，建议启用 bundled SQLite
6. 配置文件：INI 或 JSON
7. 日志：文本日志 + SQLite 操作日志表
8. 拼音索引：Rust 拼音库生成 `name_pinyin`、`name_initials`

说明：

1. 若后续团队更熟悉 React，前端可替换为 React + TypeScript，不影响后端命令层和数据库设计。
2. 数据库建议随程序发布同目录存放，避免环境依赖。

### 3.2 逻辑架构分层

系统分为四层：

1. 表现层：桌面 UI，负责表单、列表、搜索、报表展示与导入向导
2. 应用层：Tauri Command 层，负责参数校验、权限控制、事务编排
3. 领域层：会员、积分、兑换、迁移、备份等业务服务
4. 数据层：SQLite、配置文件、日志文件、备份文件

### 3.3 部署目录结构

建议目录结构如下：

```text
MemberClub/
  MemberClub.exe
  data/
    club.db
    backup/
    import/
  config/
    app.json
  logs/
    app.log
    error.log
```

目录说明：

1. `data/club.db`：主业务数据库
2. `data/backup/`：数据库备份文件
3. `data/import/`：导入缓存文件、迁移报告、预检结果
4. `config/app.json`：门店参数、备份参数、显示配置
5. `logs/`：应用运行日志、错误日志

### 3.4 运行方式

1. 双击 `MemberClub.exe` 即可启动
2. 首次启动自动创建 `data/`、`config/`、`logs/` 目录
3. 若数据库不存在，则自动初始化表结构和基础参数
4. 若数据库版本落后，则执行内置数据库迁移脚本

## 4. 功能模块设计

### 4.1 会员管理模块

职责：

1. 新增会员
2. 编辑会员
3. 查看会员详情
4. 停用会员
5. 维护拼音全拼与首字母索引

关键规则：

1. `mobile` 不设唯一约束，但必须可检索
2. `member_no` 必须唯一
3. 删除会员采用“停用”而不是物理删除
4. 会员保存时自动生成 `name_pinyin` 和 `name_initials`

### 4.2 搜索模块

职责：

1. 手机号精确与模糊搜索
2. 姓名模糊搜索
3. 拼音首字母搜索
4. 会员编号精确搜索
5. 混合关键字统一搜索

实现建议：

1. 提供单一搜索框，应用层自动识别输入类型
2. 针对 `member_no`、`mobile`、`name`、`name_initials` 建立索引
3. 结果列表展示会员摘要字段，详情区域异步加载完整历史

### 4.3 消费登记与积分模块

职责：

1. 录入消费记录
2. 按积分规则自动计算积分
3. 更新会员累计消费额、最近消费时间、当前积分
4. 生成积分流水

关键规则：

1. 积分规则为“每消费 X 元增加 1 积分”
2. 默认向下取整：`points = floor(amount / rule_amount)`
3. 消费记录、积分流水、会员余额必须同事务提交
4. 支持后续扩展手工调整和活动倍数日

### 4.4 礼品与积分兑换模块

职责：

1. 礼品目录维护
2. 积分兑换登记
3. 自动扣减积分
4. 生成兑换记录和扣减流水

关键规则：

1. 兑换前必须校验当前积分余额
2. 兑换成功后先记录业务单据，再写积分流水，再更新余额
3. 礼品名称需在兑换记录中保留快照，避免礼品后续改名影响历史单据

### 4.5 数据迁移模块

职责：

1. 导入旧版 `club.mdb` 或迁移工具输出数据
2. 预检数据质量
3. 展示预检与导入结果
4. 记录导入批次、导入明细和错误原因
5. 提供重复导入保护

关键规则：

1. 同一批数据不能重复入库
2. 导入过程允许重复执行
3. 导入前后必须保留统计比对结果
4. 必须保留旧主键到新主键映射关系

### 4.6 基础设置模块

职责：

1. 积分规则配置
2. 门店信息配置
3. 备份路径配置
4. 打印模板或导出模板配置
5. 兼容参数保留，如 `legacy_jpdj`

### 4.7 备份恢复模块

职责：

1. 手工备份
2. 自动备份
3. 恢复前保护性备份
4. 恢复结果提示和日志记录

关键规则：

1. 恢复操作必须要求管理员确认
2. 恢复前自动备份当前数据库
3. 恢复过程应校验备份文件完整性

### 4.8 报表统计模块

职责：

1. 会员总数
2. 本月新增会员数
3. 本月消费总额
4. 本月新增积分
5. 本月积分兑换次数
6. 会员消费排行

### 4.9 操作日志模块

职责：

1. 记录关键业务动作
2. 记录配置变更
3. 记录导入、恢复、停用等高风险动作

建议记录字段：

1. 操作时间
2. 操作人
3. 操作模块
4. 操作类型
5. 业务主键
6. 请求参数摘要
7. 结果状态
8. 错误信息

## 5. 关键业务流程设计

### 5.1 新增会员流程

1. 前端提交会员基础信息
2. 应用层执行字段校验
3. 生成 `member_no` 或校验用户输入的会员编号唯一性
4. 生成拼音全拼与首字母索引
5. 插入 `members`
6. 记录操作日志

### 5.2 消费登记流程

1. 根据会员编号或搜索结果定位会员
2. 读取当前积分规则 `points_rule_amount`
3. 计算本次新增积分：`floor(amount / rule_amount)`
4. 开启 SQLite 事务
5. 插入 `consumption_records`
6. 插入 `points_ledger`，`change_type=ADD`，`source_type=CONSUME`
7. 更新 `members.points_balance`、`total_spent`、`last_consume_at`
8. 写入操作日志
9. 提交事务

### 5.3 积分兑换流程

1. 读取会员当前积分与礼品配置
2. 计算本次消耗积分
3. 校验积分余额是否足够
4. 开启事务
5. 插入 `gift_redemptions`
6. 插入 `points_ledger`，`change_type=DEDUCT`，`source_type=GIFT`
7. 更新 `members.points_balance`
8. 如启用库存管理则扣减 `gifts.stock_qty`
9. 写入操作日志
10. 提交事务

### 5.4 备份流程

1. 校验数据库当前无未完成恢复任务
2. 执行 SQLite `VACUUM INTO` 或文件级备份
3. 生成带日期时间的备份文件名
4. 记录备份日志

### 5.5 恢复流程

1. 用户选择备份文件
2. 校验文件存在、扩展名、大小和 SQLite 可读性
3. 自动对当前数据库做恢复前备份
4. 替换主数据库文件
5. 重启应用或重新建立连接
6. 记录恢复日志

## 6. 数据库设计

### 6.1 数据库版本管理

建议新增数据库版本表：

```sql
CREATE TABLE IF NOT EXISTS schema_versions (
  version_no INTEGER PRIMARY KEY,
  description TEXT NOT NULL,
  applied_at TEXT NOT NULL
);
```

说明：

1. 每次数据库结构变更都需追加迁移脚本
2. 程序启动时自动检测并升级

### 6.2 核心表设计

#### 6.2.1 会员表 `members`

```sql
CREATE TABLE IF NOT EXISTS members (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  member_no INTEGER NOT NULL UNIQUE,
  name TEXT NOT NULL,
  gender TEXT,
  birth_month TEXT,
  birth_day TEXT,
  mobile TEXT,
  name_pinyin TEXT,
  name_initials TEXT,
  points_balance INTEGER NOT NULL DEFAULT 0,
  total_spent NUMERIC NOT NULL DEFAULT 0,
  last_consume_at TEXT,
  status TEXT NOT NULL DEFAULT 'ACTIVE',
  remark TEXT,
  legacy_member_id TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_members_mobile ON members(mobile);
CREATE INDEX IF NOT EXISTS idx_members_name ON members(name);
CREATE INDEX IF NOT EXISTS idx_members_initials ON members(name_initials);
CREATE INDEX IF NOT EXISTS idx_members_legacy_member_id ON members(legacy_member_id);
```

设计说明：

1. `legacy_member_id` 用于保留旧系统会员主键，支持迁移追溯和幂等
2. `status` 建议取值：`ACTIVE`、`INACTIVE`

```sql
CREATE TABLE IF NOT EXISTS migration_entity_map (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  batch_no TEXT NOT NULL,
  entity_type TEXT NOT NULL,
  legacy_pk TEXT NOT NULL,
  target_id TEXT NOT NULL,

```

### 6.3 数据一致性约束

1. `members.points_balance` 必须等于该会员积分流水累计值
2. `members.total_spent` 可由消费记录汇总校验
3. 所有影响积分的操作必须写入 `points_ledger`
4. 所有高风险操作必须写入 `operation_logs`

## 7. 迁移与幂等设计

### 7.1 导入模式

系统支持两种导入入口：

1. 直接读取 `club.mdb` 的导入模块
2. 读取外部迁移工具生成的中间数据文件，如 CSV、JSON、SQLite

建议优先实现第二种方式，降低对本机 MDB 驱动的依赖。

### 7.2 预检流程

预检阶段只读，不入正式业务表：

1. 读取源数据
2. 校验必填项、手机号、金额、积分、重复主键
3. 输出预检统计和错误清单
4. 生成 `batch_fingerprint`
5. 判断是否与历史成功批次重复

`batch_fingerprint` 生成建议：

```text
SHA256(源文件内容哈希 + 导入范围 + 数据总数 + 主键摘要)
```

### 7.3 正式导入策略

正式导入分为三步：

1. 创建 `migration_batches` 记录，状态为 `PROCESSING`
2. 按实体类型分批导入会员、消费、兑换、参数
3. 成功后更新批次状态为 `SUCCESS`，失败则为 `FAILED`

### 7.4 重复导入保护

采用两层保护：

1. 批次级保护：`migration_batches.batch_fingerprint` 唯一
2. 数据级保护：`migration_entity_map` 对 `(entity_type, legacy_pk)` 唯一

执行规则：

1. 若同一 `batch_fingerprint` 已成功导入，则禁止再次正式导入，仅允许查看历史报告
2. 若批次未成功完成，可在修复后重试
3. 重试时已导入成功的旧主键记录直接跳过，不重复写入

### 7.5 迁移映射策略

建议保留以下映射：

1. 旧会员主键 -> 新 `members.id`
2. 旧消费记录主键 -> 新 `consumption_records.id`
3. 旧兑换记录主键 -> 新 `gift_redemptions.id`

映射用途：

1. 导入幂等
2. 历史追溯
3. 问题定位
4. 数据抽样核对

### 7.6 历史积分导入策略

根据旧系统实际情况支持两种模式：

1. 余额导入模式：仅导入当前积分余额，并生成一条 `IMPORT` 类型的初始化流水
2. 流水导入模式：导入全部历史积分流水，并重新核对余额

建议：

1. 若旧系统流水质量可靠，优先导入流水
2. 若旧系统历史数据缺失严重，可先导入余额，再补历史单据

## 10. 性能设计

### 10.1 目标指标

1. 1 万会员规模内搜索响应时间不超过 1 秒
2. 消费登记、积分兑换提交时间不超过 1 秒
3. 启动时间小于 3 秒

### 10.2 优化手段

1. 关键搜索字段建立索引
2. 列表页只查询摘要字段，详情页延迟加载
3. 报表查询按月份聚合，避免全表扫描
4. 事务尽量短，避免长事务锁库
5. 启动时只加载必要配置，不预加载全量历史数据

## 11. 安全与可靠性设计

### 11.1 数据安全

1. SQLite 开启 WAL 模式，提升并发读写与崩溃恢复能力
2. 核心业务使用事务包裹
3. 恢复前自动创建保护性备份
4. 导入失败不污染正式业务数据

### 11.2 权限控制

建议角色至少分为：

1. 店员：会员、搜索、消费、兑换
2. 管理员：参数设置、数据迁移、备份恢复、操作员维护、日志查看

### 11.3 审计要求

以下动作必须审计：

1. 会员停用
2. 消费登记
3. 积分兑换
4. 手工积分调整
5. 迁移导入
6. 备份恢复
7. 参数修改

## 14. 风险与后续确认事项

1. 旧版 `club.mdb` 字段与表关系未最终确认，需在开发前补齐映射清单。
2. `JPDJ` 的真实含义可能影响积分规则或礼品规则，需优先确认。
3. 若旧系统存在大量重复会员或脏数据，需补充人工合并策略。
4. 若后续新增联网或多门店能力，应将应用层服务接口抽象为可远程化调用。

## 15. 结论

本技术方案以“单机、绿色、可迁移、可审计”为核心，采用 Tauri 2 + Rust + SQLite 的桌面本地化架构，可满足当前门店场景对稳定运行、快速检索、事务一致性和简易部署的要求。对于最关键的历史数据迁移风险，方案通过批次指纹、旧主键映射和唯一约束实现“导入可重试、同批不重复入库”的目标，可作为后续详细设计与开发实施的基础版本。
